# ANP Profile 7：附件与对象传输（修订草案）

- 文档编号：ANP-P7
- 标题：附件与对象传输
- 状态：Draft
- 版本：0.4.0
- 语言：中文
- 适用范围：本 Profile 适用于 ANP 中附件、大对象与媒体对象的互通语义，支持非加密与端到端加密，支持私聊与群聊。

---

## 1. 目的

本 Profile 定义 ANP 的附件与对象传输语义，规定：

1. 如何在私聊与群聊中表示附件；
2. 如何同时支持非加密附件与端到端加密附件；
3. 如何在 Direct Base、Group Base、Direct E2EE、Group E2EE 上承载附件清单；
4. 如何通过独立 HTTP(S) 通道上传和下载对象内容；
5. 如何通过对象定位 URI、短时下载票据和对象级加密降低链接泄露风险；
6. 如何对对象内容做完整性校验、访问控制与可选的对象级加密；
7. 如何避免把附件大字节流放入 ANP 的跨域服务调用链路或业务消息正文中转发。

本 Profile **不**定义：

- 对象存储的内部实现；
- 流媒体实时协商；
- 历史对象检索；
- 对象审核与风控策略；
- Agent 内部对象缓存、副本同步或离线下载策略；
- 对象下载票据的具体密码学签发格式。

---

## 2. 术语与规范性关键字

本文中的 **MUST**、**MUST NOT**、**REQUIRED**、**SHALL**、**SHALL NOT**、**SHOULD**、**SHOULD NOT**、**RECOMMENDED**、**NOT RECOMMENDED**、**MAY**、**OPTIONAL** 按其大写形式解释为规范性要求。

术语定义：

- **Attachment**：通过 ANP 消息引用的外部对象，其内容可以是文件、图片、音频、视频或任意字节流。
- **Object**：附件对应的实际字节内容。
- **Object Service**：负责提供对象上传、下载与访问控制的服务端点。
- **Upload Slot**：发送方为上传对象而申请的临时上传能力与元数据占位。
- **Committed Object**：已经上传完成并可被消息引用的对象。
- **Attachment Manifest**：描述对象位置、摘要、大小、媒体属性与加密属性的结构化对象。
- **Attachment Message**：以一个或多个 `attachment_manifest` 作为主要负载的 ANP 消息。
- **Object URI**：对象定位 URI 或下载 URI。
- **Object Locator**：不能直接下载内容、但可定位对象并触发票据流程的 URI。
- **Direct Download URL**：可直接下载对象的 URL，通常应为短时 URL。
- **Mirror URI**：对象镜像下载地址。
- **Download Ticket**：对象下载时使用的临时访问票据或令牌。
- **Ticket Binding**：下载票据与接收方身份、目标上下文、附件 ID 等约束的集合。
- **Object-Level Encryption**：由发送方在本地对对象内容进行的加密，而不是仅依赖传输层机密性。

## 3. 设计原则

### 3.1 消息与对象分离

ANP 中的大对象字节流 **MUST NOT** 直接嵌入 `direct.send` 或 `group.send` 的业务正文中。发送方 **SHOULD** 先把对象写入某个 Object Service，再通过消息传递 `attachment_manifest`。

### 3.2 HTTP 直连下载

附件对象 **MUST** 通过独立的 HTTP(S) 通道下载，而不是通过 ANP 的跨域服务调用链路中继对象字节。也就是说：

- ANP 的跨域服务调用仅负责转发业务消息与清单；
- 对象内容本身应由接收方通过 `object_uri` 直接向 Object Service 下载；
- 跨域服务调用链路不得作为附件字节流的常规转发通道。

### 3.3 下载链接不可绝对保密

对象定位 URI、下载 URL 或下载票据一旦被合法接收方获得，就可能被再次转发。因此，本 Profile 的安全目标不是“绝对防止别人知道链接”，而是通过以下三层手段降低风险：

1. 对象定位 URI 不可猜测、不可枚举；
2. 下载动作需要短时、可绑定身份的票据；
3. 在需要强保密时，对象内容本身使用对象级加密。

### 3.4 传输与对象加密分离

对象的安全可以来自两层：

1. **传输保护**：对象通过 HTTPS 下载；
2. **对象级加密**：发送方在本地先加密对象，再上传密文对象。

这两层可以单独存在，也可以叠加存在。

### 3.5 Manifest 复用

同一套 `attachment_manifest` 语义 **MUST** 能够同时用于：

- Direct Base（非 E2EE 私聊）
- Group Base（非 E2EE 群聊）
- Direct E2EE（E2EE 私聊）
- Group E2EE（E2EE 群聊）

差别不在清单对象本身，而在：

- 清单被谁承载；
- 清单是否被 E2EE 保护；
- `object_key` 是否能够明文出现。

### 3.6 非目标

本 Profile 不要求：

- 发送方和接收方共享同一种对象存储；
- 对象服务必须属于发送方域或接收方域；
- 所有对象都必须做对象级加密；
- 所有对象都必须经由 Group Host 提供。

## 5. 承载规则：私聊、群聊、加密、非加密

### 5.1 在 Direct Base 中承载

当附件清单通过 `direct.send` 在 `anp.direct.base.v1` 中发送时：

- `meta.profile` **MUST** 等于 `anp.direct.base.v1`；
- `meta.security_profile` **MUST** 等于 `transport-protected`；
- `meta.content_type` **MUST** 等于 `application/anp-attachment-manifest+json`；
- `body.payload` **MUST** 为 Attachment Message 对象；
- `auth.sender_proof` **MUST** 按 P3 要求存在并绑定整个 Signed Direct Payload。

### 5.2 在 Group Base 中承载

当附件清单通过 `group.send` 在 `anp.group.base.v1` 中发送时：

- `meta.profile` **MUST** 等于 `anp.group.base.v1`；
- `meta.security_profile` **MUST** 等于 `transport-protected`；
- `meta.content_type` **MUST** 等于 `application/anp-attachment-manifest+json`；
- `body.payload` **MUST** 为 Attachment Message 对象；
- `auth.actor_proof` **MUST** 按 P4 要求存在并绑定整个 Signed Group Payload；
- 成功响应 **SHOULD** 返回 `group_receipt`。

### 5.3 在 Direct E2EE 中承载

当附件清单通过 `anp.direct.e2ee.v1` 发送时：

- 外层 `meta.content_type` **MUST** 仍为 `application/anp-direct-cipher+json`（或建链首条消息对应的 init 类型）；
- 附件清单 **MUST** 作为加密前的 `Application Plaintext` 的内层业务对象出现；
- 其内层 `application_content_type` **MUST** 等于 `application/anp-attachment-manifest+json`。

也就是说，**Direct E2EE 下，Attachment Manifest 是内层内容类型，不是外层内容类型。**

### 5.4 在 Group E2EE 中承载
当附件清单通过 `anp.group.e2ee.v1` 发送时：

- 外层 `meta.content_type` **MUST** 为 `application/anp-group-cipher+json`；
- 外层 `body` **MUST** 直接承载 `group_cipher_object`，**MUST NOT** 直接出现明文附件清单；
- 附件清单 **MUST** 作为加密前的群应用明文出现；
- 其内层 `application_content_type` **MUST** 等于 `application/anp-attachment-manifest+json`。

也就是说，**Group E2EE 下，Attachment Manifest 只出现在内层明文中；外层永远表示群密文承载对象，而不是原始应用内容类型。**

### 5.5 对象级加密的使用约束

- 在 `transport-protected` 模式下，清单 **MAY** 表示未加密对象，也 **MAY** 表示发送方已加密对象；
- 在 `direct-e2ee` 或 `group-e2ee` 模式下，若对象级加密启用，则对象密钥信息 **MAY** 出现在清单中，因为清单本身已位于 E2EE 保护之内；
- 在非 E2EE 模式下，清单 **MUST NOT** 直接暴露足以解密对象的裸对象密钥，除非部署另行定义并显式允许。

## 6. HTTP 通道要求

### 6.1 下载要求

对象下载 **MUST** 使用 HTTPS。对象服务 **SHOULD** 支持：

- HTTP `GET` 下载对象；
- HTTP `HEAD` 查询对象元数据（推荐）；
- Range 请求（可选，用于断点续传）。

### 6.2 上传要求

若实现支持标准化上传控制面，则对象服务 **SHOULD** 支持：

- 先申请 Upload Slot；
- 再通过返回的 `upload_uri` 用 HTTP `PUT`、`POST` 或多段上传方式上传对象；
- 最后提交对象。

### 6.3 禁止通过跨域服务调用链路转发对象字节

ANP 的跨域服务调用 **MUST NOT** 作为附件对象字节的常规转发通道。也就是说，对象内容本身不应被重新封装进业务消息或服务到服务调用体中。

跨域服务调用链路允许转发的仅是：

- Attachment Message
- Upload/Commit 控制请求
- 下载票据请求

## 7. `attachment_manifest` 对象

### 7.1 Attachment Message 顶层结构

推荐结构如下：

```json
{
  "attachments": [
    {
      "attachment_id": "att-001",
      "filename": "report.pdf",
      "mime_type": "application/pdf",
      "size": "1048576",
      "digest": {
        "alg": "sha-256",
        "value_b64u": "BASE64URL_DIGEST"
      },
      "access_info": {
        "object_uri": "https://objects.example.com/o/123",
        "uri_mode": "locator",
        "mirror_uris": [],
        "requires_ticket": true,
        "ticket_delivery": "fetch"
      }
    }
  ],
  "caption": "附件说明",
  "primary_attachment_id": "att-001",
  "annotations": {}
}
```

规则：

- `attachments` **MUST** 存在且非空；
- `caption` **MAY** 存在；
- `primary_attachment_id` **MAY** 存在；
- `annotations` **MAY** 存在。

### 7.2 单个 `attachment_manifest`

每个 `attachment_manifest` 推荐包含以下字段：

- `attachment_id`：字符串，**MUST**
- `object_id`：字符串，**MAY**
- `filename`：字符串，**MAY**
- `mime_type`：字符串，**MUST**
- `size`：十进制字符串，**MUST**
- `digest`：对象，**MUST**
- `access_info`：对象，**MUST**
- `encryption_info`：对象，**MAY**
- `media_info`：对象，**MAY**
- `thumbnail`：对象，**MAY**
- `chunking_info`：对象，**MAY**
- `expires_at`：RFC 3339 时间字符串，**MAY**
- `annotations`：对象，**MAY**

### 7.3 `digest`

`digest` **MUST** 包含：

- `alg`
- `value_b64u`

接收方下载对象后 **MUST** 以 `digest` 校验对象完整性。

### 7.4 `access_info`
推荐字段：

- `object_uri`：字符串，**MUST**
- `uri_mode`：字符串，**SHOULD**，推荐值：
  - `locator`
  - `direct-url`
- `mirror_uris`：字符串数组，**MAY**
- `requires_ticket`：布尔值，**MAY**
- `ticket_delivery`：字符串，**MAY**，推荐值：
  - `inline`
  - `fetch`
- `control_service_did`：字符串，**MAY**；当 `requires_ticket = true` 时 **MUST**
- `download_ticket_b64u`：字符串，**MAY**
- `ticket_transport`：字符串，**MAY**，推荐值：
  - `authorization-header`
  - `custom-header`
- `headers_template`：对象，**MAY**
- `expires_at`：RFC 3339 时间字符串，**MAY**

规则：

1. `object_uri` **MUST** 为 `https://` URL；
2. `uri_mode` 默认 **SHOULD** 为 `locator`，不推荐长期公开直链；
3. 若 `uri_mode = direct-url`，则该 URL **SHOULD** 为短时 URL；
4. 若 `requires_ticket = true` 且 `ticket_delivery = fetch`，接收方 **MUST** 通过 `attachment.get_download_ticket` 获取票据；
5. 若 `requires_ticket = true` 且 `ticket_delivery = inline`，则 `download_ticket_b64u` **MUST** 存在；
6. 若 `requires_ticket = true`，`control_service_did` **MUST** 指向负责处理 `attachment.get_download_ticket` 的公开 `ANPMessageService.serviceDid`；
7. `download_ticket_b64u` **SHOULD NOT** 通过查询参数承载，推荐通过 HTTP 头承载；
8. `headers_template` 用于提示下载时需要携带的附加 HTTP 头，但接收方 **MUST** 以 Object Service 的实际要求为准。

### 7.5 `encryption_info`
推荐字段：

- `mode`：字符串，**MUST**，推荐值：
  - `none`
  - `service-managed`
  - `object-e2ee`
- `object_cipher`：字符串，**MAY**
- `object_key_b64u`：字符串，**MAY**
- `wrapped_object_key_b64u`：字符串，**MAY**
- `key_wrap_alg`：字符串，**MAY**
- `nonce_b64u`：字符串，**MAY**
- `plaintext_size`：十进制字符串，**MAY**
- `aad_binding`：对象，**MAY**

规则：

1. 当 `mode = "none"` 时，对象内容本身未做发送方本地加密；
2. 当 `mode = "service-managed"` 时，对象服务可能在服务端加密存储，但这 **不**等价于端到端加密；
3. 当 `mode = "object-e2ee"` 时，对象内容 **MUST** 由发送方在本地先加密再上传；
4. 在 `transport-protected` 模式下，清单 **MUST NOT** 直接暴露 `object_key_b64u`；
5. 在 `direct-e2ee` 或 `group-e2ee` 模式下，清单 **MAY** 包含 `object_key_b64u`，因为该清单本身受 E2EE 保护；
6. manifest 中的 `size` 与 `digest` 一律指向**上传对象字节**；当 `mode = "object-e2ee"` 时，它们指向密文字节，而 `plaintext_size` 表示原始对象大小；
7. `mime_type` 始终表示原始对象 MIME type，而不是密文对象的 MIME type。



### 7.5.1 `object-e2ee` 的 MTI 对象格式

为保证 v1 附件 E2EE 可互通，本 Profile 对 `object-e2ee` 定义如下最小强制互通格式：

```json
{
  "mode": "object-e2ee",
  "object_cipher": "chacha20-poly1305",
  "object_key_b64u": "BASE64URL_OBJECT_KEY",
  "nonce_b64u": "BASE64URL_12_BYTES",
  "plaintext_size": "1048576"
}
```

规则：

- `object_cipher` 在 v1 MTI **MUST** 固定为 `chacha20-poly1305`
- `nonce_b64u` 在 `mode = "object-e2ee"` 时 **MUST** 存在
- `object_key_b64u` 仅允许出现在 `direct-e2ee` 或 `group-e2ee` 保护下的清单中
- 上传到 Object Service 的字节流 **MUST** 是密文字节
- 接收方下载后 **MUST** 先按 `digest` 校验上传对象字节，再根据 `object_cipher`、`object_key_b64u`、`nonce_b64u` 解密
- 解密成功后，接收方 **SHOULD** 再用 `plaintext_size` 做结果长度检查

`service-managed`、自定义 key wrap、multipart 和 Range 扩展都不属于本节 MTI 主线。

### 7.6 `media_info`

适用于图片、音频、视频等媒体对象，推荐字段：

- `width`
- `height`
- `duration_ms`
- `codec`
- `language`

这些字段均 **MAY** 存在，且若出现，数值字段 **MUST** 使用十进制字符串。

### 7.7 `thumbnail`

若存在缩略图，`thumbnail` **MAY**：

- 直接是一个精简版附件清单；
- 或是一个对象引用结构。

若缩略图需要与主对象同等保密，则 **MUST** 采用与主对象相同的安全模式。

### 7.8 `chunking_info`

若对象按分块方式组织，推荐字段：

- `chunk_size`：十进制字符串，**MUST**
- `chunk_count`：十进制字符串，**MUST**
- `chunk_digests`：数组，**MAY**
- `merkle_root_b64u`：字符串，**MAY**

接收方 **SHOULD** 支持分块校验和整体校验并行使用。

## 8. 下载链接与票据安全

### 8.1 对象定位 URI 优先

`object_uri` **SHOULD** 优先表示对象定位 URI（locator），而不是长期公开可下载直链。也就是说：

- 看到 `object_uri` 不应天然意味着任何人都能直接下载；
- 合法接收方通常还需要下载票据或其它访问授权。

### 8.2 下载票据的目标

下载票据的目标不是“绝对防止合法接收方再次分享”，而是：

1. 防止第三方仅凭清单就直接下载对象；
2. 防止对象 URI 成为长期复用的公共链接；
3. 把下载授权收束到明确的接收方和上下文；
4. 为对象服务实施速率限制、审计和撤销提供抓手。

### 8.3 票据绑定

当对象服务使用下载票据时，票据 **SHOULD** 绑定以下最小上下文：

- `attachment_id`
- `object_uri` 或 `object_id`
- `requester_did`
- `security_profile`
- `message_id`（若可用）
- `target_did` 或 `group_did`（二选一，按上下文）
- `expires_at`

### 8.4 票据有效期

下载票据 **SHOULD** 是短时票据。推荐：

- 默认有效期不超过 5 分钟；
- 对高敏感对象 **MAY** 使用一次性票据；
- 若需要断点续传或多副本下载，服务端 **MAY** 放宽为有限次数票据。

### 8.5 票据传输

下载票据 **SHOULD** 通过 HTTP 头传输，**SHOULD NOT** 通过 URL 查询参数承载，以降低：

- 日志泄露；
- 代理缓存泄露；
- 浏览器历史或 Referer 泄露。

推荐：

- `Authorization` 头；或
- 服务自定义头（由 `ticket_transport` 和 `headers_template` 指示）。

### 8.6 私聊中的票据

在私聊场景中：

- 对 `transport-protected` 模式，`attachment.get_download_ticket` 的接收方身份 **SHOULD** 绑定到 `target_did`；
- 对 `direct-e2ee` 模式，若对象级加密启用，即使票据或 `object_uri` 泄露，第三方也只能得到密文对象。

### 8.7 群聊中的票据

在群聊场景中：

- 下载票据 **SHOULD** 绑定到单个请求下载的 `requester_did`，而不是单纯绑定到整个群；
- `group_did` **SHOULD** 作为附加上下文进入票据；
- 对 `group-e2ee` 模式，若对象级加密启用，即使某个群外第三方拿到链接或票据，也不应能恢复对象明文。

---



### 8.8 成员变化后的附件访问规则

v1 **不保证追溯性撤权**。

这意味着：

- 对尚未下载的对象，Object Service **SHOULD** 根据当前群状态拒绝为已被移出成员签发新的下载票据；
- 但若对象字节、下载票据或对象密钥此前已经到达合法成员，本规范 **不承诺** 事后撤回其已获得的访问能力；
- 在 `group-e2ee + object-e2ee` 组合下，新的票据签发可以阻止“以后再下载”，但不能逆转“过去已经拿到的密文或密钥”。

## 9. 控制面方法

### 9.1 总则
本节方法用于对象上传与访问控制面。它们 **不**改变 Direct Base、Group Base、Direct E2EE、Group E2EE 的发送成功语义。

这些方法默认运行在：

- `meta.profile = anp.attachment.v1`
- `meta.security_profile = transport-protected`

并采用以下统一规则：

1. `attachment.*` 控制面方法是 **service-scoped** 方法；
2. `meta.target.kind` **MUST** 为 `"service"`；
3. `meta.target.did` **MUST** 为目标公开 `ANPMessageService.serviceDid`；
4. 标准跨域路径 **MUST** 至少要求 hop / service 级认证；
5. v1 **不**再额外引入第四套终端用户 proof 作为 Object Service 的默认校验入口。

对下载票据尤其如此：Object Service 默认信任调用方服务身份对 `requester_did` 的背书，并根据请求体中的 `requester_did`、`target_did` / `group_did`、`message_id`、`security_profile` 等上下文做授权判断；它 **不**要求再直接验证一个独立的终端用户 `actor_proof`，除非部署方另有私有扩展。

### 9.2 `attachment.create_slot`

#### 9.2.1 语义

为待上传对象申请一个 Upload Slot。

#### 9.2.2 请求要求

`meta.target.kind` **MUST** 为 `"service"`，`meta.target.did` **MUST** 为目标附件控制服务的 `serviceDid`。

`body` 推荐包含：

- `attachment_id`：字符串，**MAY**
- `expected_size`：十进制字符串，**SHOULD**
- `expected_digest`：对象，**MAY**
- `mime_type`：字符串，**SHOULD**
- `filename`：字符串，**MAY**
- `intended_security_profile`：字符串，**MUST**
- `intended_target`：对象，**MAY**
- `retention_hint`：对象，**MAY**
- `object_encryption_mode`：字符串，**SHOULD**

#### 9.2.3 成功响应

成功响应 **MUST** 至少包含：

- `attachment_id`
- `slot_id`
- `upload_uri`
- `upload_method`
- `object_uri`
- `commit_token`
- `expires_at`

成功响应 **MAY** 包含：

- `multipart`
- `max_part_size`
- `required_headers`
- `uri_mode`

### 9.3 `attachment.commit_object`

#### 9.3.1 语义

通知 Object Service：对象已上传完成，可以提交为可引用对象。

#### 9.3.2 请求要求
`meta.target.kind` **MUST** 为 `"service"`，`meta.target.did` **MUST** 为目标附件控制服务的 `serviceDid`。

`body` **MUST** 至少包含：

- `attachment_id`
- `slot_id`
- `commit_token`
- `size`
- `digest`

`body` **MAY** 包含：

- `media_info`
- `chunking_info`
- `expires_at`
- `annotations`
- `encryption_info`

#### 9.3.3 成功响应

成功响应 **MUST** 至少包含：

- `committed = true`
- `attachment_id`
- `object_uri`
- `committed_at`

### 9.4 `attachment.abort_object`

#### 9.4.1 语义

终止一个未完成或不再需要的上传槽位。

#### 9.4.2 请求要求
`meta.target.kind` **MUST** 为 `"service"`，`meta.target.did` **MUST** 为目标附件控制服务的 `serviceDid`。

`body` **MUST** 至少包含：

- `attachment_id`
- `slot_id`

#### 9.4.3 成功响应

成功响应 **MUST** 至少包含：

- `aborted = true`
- `attachment_id`
- `aborted_at`

### 9.5 `attachment.get_download_ticket`

#### 9.5.1 语义

当对象访问需要票据时，接收方可通过本方法获取临时下载票据。

#### 9.5.2 请求要求
`meta.target.kind` **MUST** 为 `"service"`，`meta.target.did` **MUST** 等于对应对象的 `access_info.control_service_did`。

`body` **MUST** 至少包含：

- `attachment_id`
- `object_uri`
- `requester_did`
- `security_profile`

`body` 在私聊上下文 **MUST** 包含：

- `target_did`

`body` 在群聊上下文 **MUST** 包含：

- `group_did`

`body` **SHOULD** 包含：

- `message_id`

`body` **MAY** 包含：

- `reason`
- `one_time`：布尔值

#### 9.5.3 成功响应

成功响应 **MUST** 至少包含：

- `download_ticket_b64u`
- `expires_at`

成功响应 **SHOULD** 包含：

- `ticket_binding`
- `ticket_transport`

成功响应 **MAY** 包含：

- `headers_template`
- `download_uri_override`
- `uses_remaining`

## 10. 绑定与校验规则

### 10.1 Direct Base 绑定

在 `anp.direct.base.v1` 中：

- `meta.content_type = application/anp-attachment-manifest+json`
- `body.payload` **MUST** 为 Attachment Message
- `auth.sender_proof` **MUST** 绑定整个 Signed Direct Payload，包括该清单。

### 10.2 Group Base 绑定

在 `anp.group.base.v1` 中：

- `meta.content_type = application/anp-attachment-manifest+json`
- `body.payload` **MUST** 为 Attachment Message
- `auth.actor_proof` **MUST** 绑定整个 Signed Group Payload，包括该清单；
- `group_receipt` **SHOULD** 作为群接受结果返回。

### 10.3 Direct E2EE 绑定

在 `anp.direct.e2ee.v1` 中：

- 清单 **MUST** 位于内层 `Application Plaintext`；
- 内层 `application_content_type` **MUST** 为 `application/anp-attachment-manifest+json`；
- 若对象采用 `object-e2ee`，则 `object_key_b64u` **MAY** 出现在清单中，但必须受外层 E2EE 保护。

### 10.4 Group E2EE 绑定
在 `anp.group.e2ee.v1` 中：

- 外层 `meta.content_type` **MUST** 为 `application/anp-group-cipher+json`
- 清单 **MUST** 位于内层群应用明文；
- 内层 `application_content_type` **MUST** 为 `application/anp-attachment-manifest+json`；
- 若对象采用 `object-e2ee`，则 `object_key_b64u` **MAY** 出现在清单中，但必须受外层群 E2EE 保护。

### 10.5 下载后校验

接收方 **MUST** 在下载完成后至少校验：

- 字节长度是否与 `size` 一致；
- 摘要是否与 `digest` 一致；
- 若启用对象级加密，解密后是否成功且结果可解析。

### 10.6 失败处理

若任一关键校验失败，接收方：

- **MUST NOT** 把对象作为有效附件交付给上层业务；
- **SHOULD** 记录诊断信息；
- **MAY** 触发应用层错误或重新获取票据/对象。

## 11. 错误条件与建议错误码

本 Profile 推荐以下业务错误码：

- `anp.attachment.slot_not_found`
- `anp.attachment.slot_expired`
- `anp.attachment.commit_token_invalid`
- `anp.attachment.object_too_large`
- `anp.attachment.unsupported_mime_type`
- `anp.attachment.download_ticket_required`
- `anp.attachment.download_ticket_invalid`
- `anp.attachment.ticket_binding_mismatch`
- `anp.attachment.ticket_expired`
- `anp.attachment.digest_mismatch`
- `anp.attachment.object_unavailable`
- `anp.attachment.encryption_policy_violation`

错误响应 **SHOULD** 在 `error.data` 中提供：

- `attachment_id`
- `slot_id`
- `object_uri`
- `max_allowed_size`
- `expected_digest`

## 12. 最小互通要求

一个符合本 Profile 的实现至少 **MUST** 支持：

1. `attachment.create_slot`
2. `attachment.commit_object`
3. `application/anp-attachment-manifest+json`
4. `attachment_manifest` 的 `attachment_id`、`mime_type`、`size`、`digest`、`access_info`
5. 对象摘要校验
6. 至少一种对象获取方式（`object_uri`）
7. 独立 HTTPS 下载对象
8. 不通过 Relay 转发对象字节

若实现宣称支持更强下载控制，还 **SHOULD** 支持：

9. `attachment.get_download_ticket`
10. 票据与 `requester_did` 绑定
11. 票据短时有效期
12. 当 `requires_ticket = true` 时，在清单中提供 `access_info.control_service_did`

若实现宣称支持端到端加密附件，还 **MUST**：

13. 支持 `encryption_info.mode = object-e2ee`
14. 对 `object-e2ee` 固定支持 `chacha20-poly1305` + `nonce_b64u`
15. 在 `direct-e2ee` 或 `group-e2ee` 外层保护下传输对象密钥材料
16. 在对象下载后执行摘要与解密校验

## 附录 A（信息性）：实现建议

1. 对象内容尽量走独立 HTTPS 下载，不走 Relay；
2. `object_uri` 默认用 locator，而不是永久直链；
3. 默认要求 `requires_ticket = true`；
4. 票据默认短时有效，推荐绑定 `requester_did`；
5. 如果真的要求“链接泄露也没用”，就必须启用 `object-e2ee`。
