# ANP Profile 3：私聊基础语义（草案）

- 文档编号：ANP-P3
- 标题：私聊基础语义
- 状态：Draft
- 版本：0.2.0
- 语言：中文
- 适用范围：本 Profile 适用于 Agent 与 Agent 之间的基础私聊语义，不包含端到端加密算法本身。

---

## 1. 目的

本 Profile 定义 ANP 的私聊基础语义层，规定：

1. Agent 到 Agent 的直接消息如何表示；
2. 私聊消息的最小互通字段与内容模型；
3. 私聊消息的成功语义、幂等语义与排序语义；
4. 在非 E2EE 模式下，私聊消息如何独立运行；
5. 基于 did:wba 的发送方身份认证与跨域原发者证明如何表示；
6. Direct E2EE Profile 如何在本 Profile 的业务语义之上叠加。

本 Profile **不**定义：

- 具体 E2EE 算法；
- 设备或内部副本概念；
- 历史拉取；
- 已读状态；
- Presence；
- Agent 内部同步；
- 大对象密文格式。

## 2. 术语与规范性约定

### 2.1 规范性关键字

本文中的 **MUST**、**MUST NOT**、**REQUIRED**、**SHALL**、**SHALL NOT**、**SHOULD**、**SHOULD NOT**、**RECOMMENDED**、**NOT RECOMMENDED**、**MAY**、**OPTIONAL** 按照其大写形式解释为规范性要求。

### 2.2 术语

- **Direct Message**：一个 Agent 发往另一个 Agent 的应用层消息。
- **Sender Agent**：发起 `direct.send` 的 Agent，由外层 `params.meta.sender_did` 标识。`sender_did` 表示“谁在业务语义上发起了这条私聊消息”，后续认证、幂等、审计与策略判断都围绕该 DID 展开。
- **Recipient Agent**：`direct.send` 的目标 Agent，由外层 `params.meta.target.did` 标识。`target.did` 表示本次私聊的唯一业务接收方，用于发现目标服务和校验寻址正确性，不等于某一跳网络地址。
- **Ingress Service**：目标 Agent 对外暴露的主入口服务，用于接收直接消息。它承担协议接受、参数校验和原发者证明验证等入口职责，但不等于目标 Agent 的内部处理器。
- **Conversation**：发送方和接收方之间的应用层会话上下文；是否存在稳定 `conversation_id` 由业务层决定。若实现使用 `conversation_id`，它只用于把多条消息归并到同一应用上下文，不承载设备路由或安全状态。
- **Accepted**：消息已被目标 Agent 的入口服务接受并进入其处理边界。它表示协议层成功，不等于上层应用已处理或已向最终用户展示。
- **Rejected**：消息在协议层被拒绝，未进入目标 Agent 的处理边界。
- **Sender Proof**：由发送方 Agent 基于 did:wba JSON 承载认证生成的应用层签名证明，用于证明某条 `direct.send` 确由 `sender_did` 对应私钥持有者发起。在线协议中它通常位于 `params.auth.sender_proof`，供最终接收方直接验证业务原发者身份。
- **Hop Authentication**：某一跳服务到服务、Agent 到服务之间的传输层/请求层身份认证。Hop Authentication 可以变化，但不得替代 Sender Proof。
- **Logical Target URI**：为使应用层签名能够跨中继保持稳定，由本 Profile 定义的逻辑目标 URI；它绑定 ANP 的业务目标，而不是某一跳的具体 HTTP URL。在线协议中它由 `target.did` 派生，供 `Signature-Input` 中的 `@target-uri` 使用。

## 3. 设计原则

### 3.1 Agent 是协议终点

ANP Direct Messaging 的协议终点是 Agent，而非设备、终端、会话副本或内部执行单元。

只要目标 Agent 的入口服务接受了该消息，本 Profile 的投递使命即告完成。目标 Agent 内部如何同步到多少个副本、多少个执行器或多少个设备，不属于本 Profile 的互通范围。

### 3.2 Base 语义与 Security Overlay 分离

本 Profile 仅定义业务语义与消息语义；若后续叠加 Direct E2EE Profile，则：

- 业务动作仍然是 `direct.send`；
- 私聊对象仍然是 `sender_did -> recipient_did`；
- 内容类型和应用语义仍由本 Profile 定义；
- 额外的密码学对象、会话状态与安全绑定由 E2EE Profile 定义。

### 3.3 最小成功语义

本 Profile 中，一次 `direct.send` 的成功不意味着：

- 目标 Agent 已处理应用负载；
- 目标 Agent 已持久化该消息；
- 目标 Agent 已转发到内部任意副本；
- 目标 Agent 已向任何人类用户展示该消息。

本 Profile 中，一次 `direct.send` 的成功仅表示：

- 目标 Agent 的入口服务已接受该消息；
- 该消息已进入目标 Agent 的协议处理边界。

### 3.4 非目标

本 Profile **不**尝试提供：

- 端到端机密性；
- 多设备一致性；
- 全局严格顺序；
- 历史可重放语义；
- 设备级送达证明。



### 3.5 原发者证明与跳级认证分离

在跨域场景中，`direct.send` 至少可能涉及两类认证：

1. **原发者证明（Sender Proof）**：由 `sender_did` 对应私钥生成，证明“这条业务消息是发送方 Agent 发出的”；
2. **跳级认证（Hop Authentication）**：由当前连接的一跳调用方生成，证明“本次网络请求是由哪个服务或哪个 Agent 提交到下一跳的”。

本 Profile 要求：

- `direct.send` 的业务原发者证明 **MUST** 可跨转发链保持稳定；
- 任意一跳的服务级认证 **MAY** 变化；
- 接收方 **MUST NOT** 仅凭上游服务的身份，就推定 `meta.sender_did` 的业务原发者身份成立。

## 4. Profile 标识与依赖

### 4.1 Profile 名称

本 Profile 的标准名称为：

`anp.direct.base.v1`

### 4.2 依赖关系

本 Profile **MUST** 依赖以下 Profile：

- `anp.core.binding.v1`
- `anp.identity.discovery.v1`

### 4.3 安全模式

本 Profile 作为独立运行的基础私聊 Profile 时：

- `meta.profile` **MUST** 等于 `anp.direct.base.v1`
- `meta.security_profile` **MUST** 等于 `transport-protected`

补充说明：

- `meta.profile` 用于声明“本次请求按照哪个业务 Profile 解释 `body` 与方法语义”；接收方据此决定字段约束和处理路径。
- `meta.security_profile` 用于声明“本次请求要求采用哪一种安全语义”；接收方与中间服务都不得静默改写或降级该字段。

若后续某 Direct E2EE Overlay 复用本 Profile 的业务动作与应用语义，则：

- 对应 Profile **MUST** 明确说明如何在 `direct.send` 语义之上叠加安全对象；
- 接收方 **MUST NOT** 在未显式协商的情况下，从高安全模式静默降级到 `transport-protected`；
- 当 `meta.profile = "anp.direct.e2ee.v1"` 时，本 Profile 中关于 `auth.sender_proof` 的强制要求由 Direct E2EE Profile 覆盖。

## 5. 直接寻址与会话模型

### 5.1 目标类型

`direct.send` 的 `meta.target.kind` **MUST** 为 `"agent"`。

`meta.target.kind` 用于声明目标实体类别；在 Direct Base 中它固定表示“本次消息面向一个 Agent 协议主体”。接收方首先依据该字段判断本请求是否仍属于私聊语义。

`meta.target.did` **MUST** 是一个可解析的 `agent_did`。

`meta.target.did` 用于声明本次 `direct.send` 的唯一业务目标：发送方依据它做 DID 解析与服务发现，接收方依据它校验“这条请求是否确实发给了我所代表的目标 Agent”。

若 `target.kind` 不是 `"agent"`，接收方 **MUST** 返回 `anp.invalid_params_shape` 或更具体的业务错误。

### 5.2 发送方与接收方

一次 `direct.send`：

- **MUST** 只有一个发送方 Agent；
- **MUST** 只有一个接收方 Agent；
- **MUST NOT** 直接面向多个目标 Agent 广播。

若调用方希望同时向多个 Agent 发送相同内容，**MUST** 发送多次独立的 `direct.send`。

### 5.3 `conversation_id`

本 Profile **MAY** 使用 `conversation_id` 表示发送方与接收方之间的应用层上下文。

`conversation_id` 的主要用途是把属于同一业务会话、线程或聊天视图的多条消息关联起来，便于上层应用做归并、展示和 best-effort 顺序处理；它不是寻址字段，也不是会话密钥 ID。

若存在 `conversation_id`：

- 它 **MUST** 是字符串；
- 它 **MUST NOT** 被解释为设备、会话副本或内部执行器标识；
- 它 **MAY** 由发送方生成，或由双方业务逻辑协商确定；
- 它 **MUST NOT** 作为唯一安全上下文或防重放标识。

### 5.4 Agent 内部实现不可见

接收方的以下内部细节 **MUST NOT** 暴露为线协议互通语义：

- 内部队列数；
- 内部副本数；
- 内部工作流节点数；
- 内部设备数；
- 内部转发路径。

## 6. 内容模型

### 6.1 内容类型

`direct.send` 的 `meta.content_type` **MUST** 存在。

`meta.content_type` 表示本次消息原始应用负载的解释方式。接收方应先依据它判断 `body` 中应使用 `text`、`payload` 还是 `payload_b64u`，再决定是否支持该负载类型。

本 Profile 最小互通 **MUST** 支持以下内容类型：

- `text/plain`
- `application/json`
- `application/anp-attachment-manifest+json`

实现方 **MAY** 支持更多内容类型，但：

- 未注册或未协商的内容类型 **SHOULD** 被拒绝；
- 接收方 **MUST** 对不支持的内容类型返回 `anp.unsupported_content_type`。

### 6.2 `body` 结构

`direct.send` 的 `body` **MUST** 是对象，并可包含以下字段：

#### 6.2.1 `conversation_id`

- 类型：字符串
- 要求：**MAY**
- 语义：应用层会话上下文标识
- 使用方式：同一业务会话内应保持稳定；当业务上进入新的会话或线程上下文时再变化。接收方通常只将其用于会话归并、排序提示或 UI 组织，而不把它当作幂等键或安全上下文。

#### 6.2.2 `reply_to_message_id`

- 类型：字符串
- 要求：**MAY**
- 语义：表示该消息回复的上一条消息
- 使用方式：填写被回复消息的 `message_id`，用于建立引用关系、线程关系或 UI 跳转关系；它不参与幂等判定，也不能替代 `message_id` 表示当前消息自身。

#### 6.2.3 `annotations`

- 类型：对象
- 要求：**MAY**
- 语义：发送方自定义的应用层附加元数据
- 使用方式：适合承载展示标签、轻量扩展属性或上层业务提示；除非双方另有明确约定，不应用它偷偷承载会改变授权、安全判定或路由判定的隐藏控制语义。
- 规则：
  - 中继服务 **MUST NOT** 擅自改写；
  - 未识别字段 **MAY** 被接收方忽略。

#### 6.2.4 `text`

- 类型：字符串
- 要求：当 `content_type = "text/plain"` 时 **MUST**
- 使用方式：用于承载用户可读的原始文本内容。发送方应直接放入文本本身，而不是再包一层 JSON 字符串或其它二次封装格式。
- 规则：
  - 使用 `text` 时，`payload` 与 `payload_b64u` **MUST NOT** 同时出现。

#### 6.2.5 `payload`

- 类型：JSON 对象
- 要求：当 `content_type` 为 JSON 类型时 **MUST**
- 使用方式：用于承载结构化应用对象；对象内部字段名和结构由具体上层业务语义决定，接收方应依据 `content_type` 与上层约定解释该对象。
- 规则：
  - `payload` **SHOULD** 直接表示应用对象；
  - **MUST NOT** 采用“JSON 字符串中再嵌 JSON”的双重序列化。

#### 6.2.6 `payload_b64u`

- 类型：字符串
- 要求：当内容为二进制扩展或私有扩展对象时 **MAY**
- 使用方式：用于承载无法自然表达为 JSON 对象的二进制负载。接收方在解码前应先依据 `content_type` 判定其业务解释方式，而不是盲目按某种私有格式解析。
- 规则：
  - 必须为无填充 base64url；
  - 使用 `payload_b64u` 时，`text` 与 `payload` **MUST NOT** 同时出现。

### 6.3 负载字段互斥规则

在 `direct.send` 中，`text`、`payload`、`payload_b64u` 三者中：

- **MUST** 恰好出现一个；
- 若出现多个，接收方 **MUST** 拒绝请求；
- 若三者均不存在，接收方 **MUST** 拒绝请求。


### 6.4 `auth` 对象

当 `meta.profile = "anp.direct.base.v1"` 且 `meta.security_profile = "transport-protected"` 时，`direct.send` 的 `params` **MUST** 包含 `auth` 对象。

`auth` 用于承载与本次业务请求绑定的原发者证明。它位于 `params` 顶层而不放入 `body`，目的是把“认证语义”与“应用负载”分离，同时避免 `auth` 自身进入 `contentDigest` 形成自引用。

当 `meta.profile = "anp.direct.e2ee.v1"` 时，是否需要、如何表示以及如何验证原发者证明，由 Direct E2EE Profile 明确定义；本节不再单独强制。

推荐结构如下：

```json
{
  "auth": {
    "scheme": "anp-rfc9421-sender-proof-v1",
    "sender_proof": {
      "contentDigest": "sha-256=:BASE64_SHA256_DIGEST:",
      "signatureInput": "sig1=(\"@method\" \"@target-uri\" \"content-digest\");created=1733402096;expires=1733402156;nonce=\"abc123\";keyid=\"did:wba:example.com:user:alice:e1_<fingerprint>#key-1\"",
      "signature": "sig1=:BASE64_SIGNATURE:"
    }
  }
}
```

字段要求：

- `auth.scheme` **MUST** 等于 `anp-rfc9421-sender-proof-v1`；该字段用于声明 `auth` 对象的解释规则和验证流程版本，接收方应先根据它决定如何解析 `sender_proof`。
- `auth.sender_proof` **MUST** 存在；它是实际的原发者证明对象，用于把当前 `direct.send` 与 `meta.sender_did` 绑定起来。
- `auth.sender_proof.contentDigest`、`auth.sender_proof.signatureInput`、`auth.sender_proof.signature` 的语义 **MUST** 与 did:wba 规范第 4 节的 JSON 承载认证一致；
- `auth` 本身 **MUST NOT** 参与 `contentDigest` 的计算。

补充说明：

- `params.auth.scheme` 描述的是与当前业务消息绑定的原发者证明格式；
- 它与 P2 `ANPMessageService.authSchemes` 所描述的 caller → service 调用认证能力 **不是同一命名空间**。

其中各字段的使用方式如下：

- `contentDigest`：对 Signed Direct Payload 计算得到的内容摘要，接收方据此确认被签名的业务对象没有被篡改。
- `signatureInput`：声明签名覆盖了哪些组件，以及 `created`、`expires`、`nonce`、`keyid` 等参数；接收方据此重建验证上下文。
- `signature`：对 `signatureInput` 所声明组件的签名值，接收方使用 `keyid` 指向的验证方法完成签名校验。
- `created`：签名创建时间，用于时间窗口校验。
- `expires`：签名失效时间，用于限制证明的可重放窗口。
- `nonce`：一次性随机值或挑战值，接收方可结合本地缓存降低重放风险。
- `keyid`：DID URL，指向实际用于验证 `sender_proof` 的验证方法。

### 6.5 Signed Direct Payload

`auth.sender_proof.contentDigest` **MUST** 绑定一个不包含 `auth` 的标准业务对象，称为 **Signed Direct Payload**。

Signed Direct Payload 的规范化结构如下：

```json
{
  "method": "direct.send",
  "meta": {
    "anp_version": "1.0",
    "profile": "anp.direct.base.v1",
    "security_profile": "transport-protected",
    "sender_did": "did:wba:...",
    "target": {
      "kind": "agent",
      "did": "did:wba:..."
    },
    "operation_id": "op-...",
    "message_id": "msg-...",
    "content_type": "text/plain",
    "created_at": "2026-03-29T12:00:00Z"
  },
  "body": {
    "...": "..."
  }
}
```

其中字段的使用方式如下：

- `method`：固定为 `direct.send`，用于防止同一份 `body` 被挪用到其它业务方法。
- `meta`：绑定发送方、目标、幂等标识、消息标识、内容类型和安全模式等业务边界；接收方验证摘要时应把它视为“这条消息在什么上下文下成立”。
- `body`：绑定实际的应用负载和会话上下文；只要 `conversation_id`、`reply_to_message_id`、`annotations`、`text`、`payload` 或 `payload_b64u` 中任一内容变化，`contentDigest` 都应随之变化。

规则如下：

- 发送方 **MUST** 使用 RFC 8785 JCS 对该对象进行规范化序列化后再计算 `contentDigest`；
- Signed Direct Payload 中缺省的可选字段 **MUST** 被直接省略，**MUST NOT** 使用 `null`、空字符串或其它占位值替代；
- 规范化后的字节序列 **MUST** 按 UTF-8 编码计算摘要；
- `trace_id`、上游服务路由信息、服务端本地附加字段 **MUST NOT** 进入 Signed Direct Payload；
- `reply_to_message_id`、`conversation_id`、`annotations` 等若存在于 `body`，则会被该摘要一并绑定。

### 6.6 ANP-WBA 应用层签名组件映射

当 did:wba 认证信息承载在 ANP JSON-RPC 请求中时，为保证签名可跨服务转发保持稳定，本 Profile 对 `Signature-Input` 中的关键组件作如下应用层映射：

- `@method`：映射为 ANP 业务方法名 `direct.send`；
- `@target-uri`：映射为逻辑目标 URI `anp://agent/<pct-encoded target.did>`；
- `content-digest`：映射为 Signed Direct Payload 的摘要值。

说明：

- 上述 `@target-uri` 是应用层逻辑目标，而不是某一跳具体 HTTP URL；
- 如果同一业务消息经过发送方 Home Service、Relay Service 或接收方 Ingress Service 转发，`@target-uri` **MUST** 仍然保持为同一个逻辑目标 URI；
- 若底层确实使用 HTTP 传输，各跳之间 **MAY** 另外再使用一次基于真实 HTTP `@method` / `@target-uri` 的 hop-level did:wba 认证，但该 hop-level 认证 **MUST NOT** 替代 `auth.sender_proof`。

## 7. 标准方法

### 7.1 `direct.send`

#### 7.1.1 语义

`direct.send` 表示发送方 Agent 向目标 Agent 发送一条直接消息。

#### 7.1.2 请求要求

以下请求要求仅适用于 `meta.profile = "anp.direct.base.v1"` 且 `meta.security_profile = "transport-protected"` 的 Base 模式直发消息。

一个合规的 `direct.send` 请求 **MUST** 满足：

1. `method = "direct.send"`
2. `meta.profile = "anp.direct.base.v1"`
3. `meta.security_profile = "transport-protected"`
4. `meta.sender_did` **MUST** 存在
5. `meta.target.kind = "agent"`
6. `meta.target.did` **MUST** 存在
7. `meta.operation_id` **MUST** 存在
8. `meta.message_id` **MUST** 存在
9. `meta.content_type` **MUST** 存在
10. `body` **MUST** 满足本 Profile 的内容互斥规则
11. `auth.scheme` **MUST** 等于 `anp-rfc9421-sender-proof-v1`
12. `auth.sender_proof` **MUST** 存在并绑定 Signed Direct Payload

#### 7.1.3 成功响应

成功响应的 `result` **MUST** 至少包含：

- `accepted`：布尔值，且 **MUST** 为 `true`
- `message_id`：与请求中的 `meta.message_id` 一致
- `operation_id`：与请求中的 `meta.operation_id` 一致
- `target_did`：目标 Agent DID
- `accepted_at`：RFC 3339 时间字符串

上述字段的使用方式如下：

- `accepted`：表示消息已进入目标入口服务的协议处理边界；它不表示最终用户已收到、已读或已处理。
- `message_id`：用于让调用方把成功响应与应用层消息一一对应，并在必要时做消息级去重。
- `operation_id`：用于让调用方把本次响应与幂等操作关联起来，并据此停止针对同一操作的重试。
- `target_did`：表示最终被该次成功响应确认接受的业务目标，便于调用方核对寻址结果。
- `accepted_at`：表示目标入口服务接受该消息的时间点，用于审计、重试窗口判断或上层展示。

标准成功响应只表示：**最终目标 Agent 的 ingress service** 已验证必要安全条件并接受该消息。  
中继、Home Service 或其它上游服务 **MUST NOT** 自行生成与该语义等价的标准成功响应。

成功响应 **MAY** 包含：

- `conversation_id`

其中：

- `conversation_id` 若返回，仅表示接收方本地接受或映射后的会话上下文；它 **MUST NOT** 被解释为改写了原始、已签名绑定的业务消息内容。

#### 7.1.4 失败条件

在以下情况下，接收方 **MUST** 拒绝 `direct.send`：

- 目标 DID 不存在或不可达；
- 当前连接的一跳调用方未通过本 hop 的调用方认证，或当前连接无权提交/转发该请求；
- 内容类型不被支持；
- 请求体不符合内容互斥规则；
- 目标策略禁止该消息；
- 请求违反安全模式要求；
- `auth.sender_proof` 缺失、无效、过期或疑似重放；
- `auth.sender_proof` 中 `keyid` 所属 DID 与 `meta.sender_did` 不一致。

### 7.2 `direct.incoming`（可选 Notification）

#### 7.2.1 语义

`direct.incoming` 是一个**可选 push capability**，用于已建立双向连接的场景中，由某 ANP Endpoint 向目标 Agent 推送一条已被目标 ingress 接受的直接消息。  
它用于传输便利，不属于 P3 的最小互通能力。

#### 7.2.2 使用范围

- 该方法 **MAY** 使用；
- 它是传输便利能力，不改变 `direct.send` 的成功语义；
- 即使不支持 `direct.incoming`，实现仍然可以是本 Profile 合规实现。

#### 7.2.3 约束

- `direct.incoming` **MUST** 作为 Notification 发送；
- 接收方 **MUST NOT** 对其返回 JSON-RPC Response；
- 一旦实现该能力，发送方 **MUST** 使用标准 Notification envelope；
- `params` **MUST** 包含：
  - `meta`
  - `body`
- `params.meta.profile` **MUST** 等于 `anp.direct.base.v1`
- `params.meta.security_profile` **MUST** 等于与原始消息一致的安全模式
- `params.meta.target.kind` **MUST** 为 `"agent"`
- `params.meta.target.did` **MUST** 等于通知接收方 DID
- `params.meta.sender_did` **MUST** 等于原始业务消息的逻辑发送主体 DID
- `params.body` **MUST** 承载与原始消息一致的业务负载；
- `params.auth.sender_proof` 若存在，**MUST** 为原始 `sender_proof` 的无损转发副本，而不是中间服务重写后的新证明。

## 8. 投递、幂等与排序语义

### 8.1 接受语义

一次 `direct.send` 返回成功，仅表示：

- 目标 Agent 的入口服务已接受该消息；
- 发送方可停止对同一 `operation_id` 的重试，除非本地策略另有要求。

### 8.2 幂等与去重

接收方 **MUST** 基于以下最小集合进行幂等判断：

- `sender_did`
- `target.did`
- `method`
- `operation_id`

对于消息级重复，接收方 **SHOULD** 进一步基于：

- `sender_did`
- `target.did`
- `message_id`

进行重复识别。

实现幂等时，接收方 **MUST** 先检查上述幂等键是否已命中已成功请求；若已命中，则 **MUST** 直接返回同一结果或等价结果。  
只有在未命中幂等记录时，接收方才继续检查其它前置条件或策略冲突。

### 8.3 排序语义

本 Profile **不**提供全局严格顺序保证。

对于直接消息：

- 实现方 **MAY** 在同一 `conversation_id` 内做 best-effort 顺序处理；
- 但跨域互通层 **MUST NOT** 假设网络或服务会天然提供严格 FIFO；
- 若后续安全 Overlay 需要更强排序约束，必须由 Overlay 自身定义计数器或会话态机制。

### 8.4 重试

发送方在重试同一消息时：

- `operation_id` **MUST** 保持不变；
- `message_id` **MUST** 保持不变；
- 业务负载 **MUST** 保持语义等价。

若重试时负载语义改变，则发送方 **MUST** 视为新的消息与新的操作。

## 9. 安全与策略

### 9.1 传输保护要求

本 Profile 在独立运行时，**MUST** 依赖经过认证的安全传输层。未受保护的裸 HTTP、裸 WebSocket 或其它未认证信道 **MUST NOT** 使用。

### 9.2 `direct.send` 的发送方身份认证

对于任何在 `meta.profile = "anp.direct.base.v1"` 下声明了 `meta.sender_did` 的 `direct.send`：

- 发送方 **MUST** 提供 `auth.sender_proof`；
- 接收方入口服务 **MUST** 按 did:wba 第 3 节 / 第 4 节的验证规则验证 `auth.sender_proof`；
- 接收方 **MUST** 验证 `auth.sender_proof` 中 `keyid` 所属 DID 与 `meta.sender_did` 一致；
- 接收方 **MUST** 验证 `keyid` 指向的验证方法被 DID 文档的 `authentication` 关系授权；
- 对路径型 `e1_` DID，接收方 **MUST** 按 did:wba 规范验证 DID 绑定关系，不得在 `authentication` 中任意选择另一个 Ed25519 key 替代。

### 9.3 `sender_proof` 的验证步骤

接收方对 `auth.sender_proof` 的最小验证步骤如下：

1. 从 `signatureInput` 中提取 `created`、`expires`、`nonce`、`keyid` 等参数；
2. 按本 Profile 第 6.6 节重建应用层 `@method`、`@target-uri` 与 `content-digest`；
3. 按本 Profile 第 6.5 节重建 Signed Direct Payload，并验证 `contentDigest` 是否匹配；
4. 解析 `keyid` 指向的 DID 与验证方法，读取 DID 文档；
5. 按 did:wba 规则验证验证方法存在、被 `authentication` 授权、签名算法匹配；
6. 检查 `created` / `expires` 是否在允许窗口内；
7. 检查 `nonce`、`operation_id`、`message_id` 的重放防护策略；
8. 若任一步骤失败，**MUST** 拒绝请求。

最小重放防护规则如下：

- 接收方 **MUST** 对 `(sender_did, target.did, method, operation_id)` 执行幂等处理；
- 接收方 **SHOULD** 在 `expires` 窗口内缓存 `(keyid, nonce, contentDigest)` 以识别异常重放；
- 若 `operation_id` 相同但 Signed Direct Payload 不同，接收方 **MUST** 返回幂等冲突；
- 若出现完全相同的已签名 payload 重放，接收方 **MAY** 返回原结果或等价结果，但 **MUST NOT** 将其视为新的独立成功投递；
- `meta.created_at` 表示业务创建时间，不是安全时间窗；重放窗口校验 **MUST** 以 `signatureInput.created` / `expires` 为准。

### 9.4 跨域转发与上游服务认证

在跨域部署中，业务消息可能经过发送方 Home Service、Relay Service 或接收方 Ingress Service。此时：

- 原始 `auth.sender_proof` **MUST** 随消息一起转发，且 **MUST NOT** 被中间服务重写；
- 接收方目标域 **MUST** 直接验证 `auth.sender_proof`，而 **MUST NOT** 仅凭上游服务身份就接受 `meta.sender_did` 的业务归属；
- 各服务跳之间 **MUST** 另外执行服务级身份认证；服务级认证可以采用 did:wba 第 3 节的 HTTP 头承载方式，或采用 ANP Federation and Relay Profile 的等价机制；
- 上游服务 **MAY** 在本地先验证 `auth.sender_proof` 再转发，但其本地验证成功 **MUST NOT** 免除目标域的独立验证责任。

### 9.5 Access Token 优化

基于 did:wba 的 access token 流程 **MAY** 用于优化重复请求，例如：

- `sender_did` 到本域 Home Service 的重复调用；
- 可信任服务之间的重复调用。

但对 `direct.send` 而言：

- access token **MUST NOT** 替代 `auth.sender_proof` 作为跨域原发者证明；
- sender-constrained access token **SHOULD** 优先于普通 Bearer token；
- 即使服务端接受 access token，仍 **SHOULD** 保留与业务消息绑定的原发者证明，以便目标域独立验证。

### 9.6 安全模式升级与降级

若某 Agent 的策略要求 `direct-e2ee`：

- 发送方 **MUST** 使用对应的 Direct E2EE Profile；
- 接收方收到 `transport-protected` 的 `direct.send` 时 **MUST** 拒绝；
- 接收方 **MUST NOT** 在未显式协商的情况下静默接受较弱安全模式。

### 9.7 与 Overlay 的绑定点

后续 Direct E2EE Overlay **SHOULD** 至少将以下字段纳入受认证上下文：

- `sender_did`
- `target.did`
- `message_id`
- `conversation_id`（若存在）
- `content_type`
- `security_profile`
- `auth.sender_proof.contentDigest` 或等价的原发者证明摘要

## 10. 隐私注意事项

### 10.1 最小元数据原则

发送方 **SHOULD** 只发送实现互通所必需的最小元数据。

### 10.2 不暴露内部拓扑

任何实现 **MUST NOT** 通过本 Profile 暴露：

- 内部设备数量；
- 内部副本数量；
- 内部工作流拓扑；
- 内部投递路径。

### 10.3 中继最小知情

中继或入口服务 **SHOULD** 仅检查协议互通所必需的最小字段；
若应用负载不需要被其理解，则 **SHOULD** 将其视为不透明负载。

## 11. Profile 特定错误（推荐）

在沿用 ANP Core 公共错误模型的前提下，本 Profile 推荐以下 `anp_code`：

| `code` | `anp_code` | 含义 |
|---|---|---|
| 2000 | `direct.recipient_unreachable` | 目标 Agent 暂不可达 |
| 2001 | `direct.policy_violation` | 目标策略拒绝该消息 |
| 2002 | `direct.invalid_payload_shape` | 私聊负载结构非法 |
| 2003 | `direct.conversation_conflict` | 会话上下文冲突 |
| 2004 | `direct.security_mode_required` | 目标要求更高安全模式 |
| 2005 | `direct.invalid_sender_proof` | 发送方原发者证明无效、过期或缺失 |
| 2006 | `direct.sender_did_mismatch` | `meta.sender_did` 与 `keyid` 所属 DID 不一致 |
| 2007 | `direct.sender_proof_replayed` | 发送方原发者证明疑似重放 |

这些错误码为推荐扩展；实现方 **MAY** 返回更通用的 ANP Core 错误码，但 **SHOULD** 在 `error.data.anp_code` 中保留可机器识别的语义。

## 12. 最小互通要求

一个符合本 Profile 的实现至少 **MUST** 支持：

1. `direct.send`
2. `target.kind = "agent"`
3. `text/plain`
4. `application/json`
5. `application/anp-attachment-manifest+json`
6. `text` / `payload` / `payload_b64u` 的互斥规则
7. `message_id` 与 `operation_id` 的幂等处理
8. `auth.sender_proof` 的提交与验证
9. 本 Profile 定义的 Signed Direct Payload 规范化与摘要计算规则
10. 基于安全传输的运行方式
